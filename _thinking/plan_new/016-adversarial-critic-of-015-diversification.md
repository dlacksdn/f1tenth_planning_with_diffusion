## 0. 용어 풀이 (본문 전에 5개만)

| 용어 | 한 줄 뜻 |
|---|---|
| **BC (scale=0)** | value 없이 prior가 데이터를 그대로 흉내. 데이터에 없는 상태에선 무력. |
| **covariate shift** | 닫힌 루프에서 작은 오차가 쌓여 **데이터 밖 상태**로 표류 → 발산. |
| **복구 covariate** | "라인에서 벗어났을 때 되돌아오는 행동" 데이터. 015가 "이게 없다"고 주장. |
| **under-determination (식별불가)** | 같은 관측(둘 다 0/10)을 **두 개 이상의 원인**(covariate vs obs)이 똑같이 설명 → 데이터만으론 못 가림. |
| **rail saturation(조향 포화)** | 조향이 물리한계(±0.4189 rad)에 붙음. 더 꺾을 권한이 0 = 복구 여력 없음. |

비유로 한 번 더: BC는 "외운 영상 재생"이고, 015의 처방(다양화 수집)은 "더 다양한 영상을 외우게 하자"는
것이다. 이 검수의 핵심 질문 두 개 — **(1) 영상이 부족해서 못 외우는 게 맞나, 아니면 차의 눈(obs)이나
차의 한계주행(margin0)이 문제인가?(진단)** **(2) 한계주행 차로 "복구 영상"을 찍을 수 있긴 한가?(처방)**

## 1. 총평 · 판정

### 판정: **조건부 채택 (Conditional Accept)** — 진단 일부 강등 + 계획 게이트 순서 재배열 필수

015는 세 가지가 동시에 사실이다:

1. **수치는 정직·정확하다.** 015 §2·§7의 거의 모든 실측을 동결 npz로 **2회 독립 재현**했고 전부 일치한다
   (§6 재계산표). 특히 품질지표(rail포화 7.76%/1.10%, lateral-demand 2.258/0.783, corr −0.032/0.000,
   CV 0.191%/0.518%, 시작 pose std≈0, frame0 lidar byte-동일)와 V1(시작RTG·페널티스윕)이 정확하다.
   **015는 데이터를 지어내지 않았다.** 비판은 수치가 아니라 **진단 논리·계획 순서**로 좁혀진다.

2. **결론1(마진≠완주)은 견고, 결론2(병목=복구 covariate)는 과잉해석이다.** cap5(큰 마진)도 BC 완주
   0/10이라는 통제실험은 "마진이 완주 원인"을 깔끔히 기각한다(013 A4형 falsification, 확증). 그러나
   거기서 "**복구 covariate 부재가 主원인**"이라는 양성 진단으로 넘어간 것은 **013 A5가 이미 '과잉해석'
   이라 판정한 진단(012 §3c "데이터 얇음")을 단어만 바꿔(얇음→covariate) 재단정**한 것이다(013:73,
   139-144). "둘 다 0/10"은 covariate 가설과 obs 가설을 **구분하지 못한다(under-determination)** — 015는
   obs 갈래를 §5·Phase4로 격리만 했지 배제하지 못했다.

3. **핵심 처방(margin0 정책에 노이즈 주입)은 메커니즘상 작동하기 어렵고, 게이트 순서가 정보가치의
   역순이다.** cap10은 7.76% step에서 조향을 **하드 물리한계(±0.4189)에 명령**하므로(f1tenth_env.py:35,
   322), 포화 구간에서 노이즈를 줘봐야 한계방향=clip 무효·반대방향=표류 악화 → **"복구 데모"를 만들
   권한 자체가 없다**(D4). 또 covariate·obs가 식별불가인데도 covariate가 참일 때만 보상받는 **비대칭
   베팅**(Phase1 다양화수집, 어댑터 수정 필요)을 먼저 두고, **동일 비용으로 한 갈래를 배제하는 저비용
   obs 프로브(env var flip 1판)를 Phase4로 미뤘다**(D7).

> 한 줄: **015를 죽이진 않는다.** 결론1과 "현 데이터로는 부족하다"는 방향은 옳다. 단 **결론2를
> '입증된 결론'에서 '미입증 작업가설'로 강등**하고, **계획을 "obs프로브 선행 → 갈래 확정 → (covariate면)
> DAgger/중간캡 자연수집, 노이즈주입은 폐기 → value"로 재배열**하라. 그리고 **011이 BC 완주를 "중간
> 확률"로 베팅했다가 012가 0/10으로 그 대전제를 반증했으므로**, 이제 다양화 fix는 "stretch 보너스"가
> 아니라 **보고서의 기저 산출물(baseline 초과 완주) 전체가 걸린 高위험 베팅**이다 — 015는 이 격상을
> 과소 반영했다(E).

015의 미덕도 적시한다: **자기-critic(§7 D1~D10)이 비정상적으로 정직하다.** D1은 "반증설계 없이
covariate로 단정하면 012 A5형 과잉해석"이라 스스로 적었고, D4·D6·D7·D9도 약점을 표기했다. **문제는
본문 §0/§3 헤더의 단정 어조("진짜 공통병목 = 복구 covariate 부재")가 D1 단서와 모순되고, 계획이 자기가
적은 D7을 따르지 않는다는 점**이다. 즉 015는 "옳게 의심하고 틀리게 베팅"했다.

## 4. D1~D10 개별 판정

> 판정어: **확증**(015 우려/주장이 1차소스로 옳음) / **반박**(틀림) / **부분반박** / **검증불가** /
> **과잉해석**(방향 맞되 단정 과함).

**D1 — "복구 covariate가 병목" 입증됐나? → 과잉해석 (단 자기표기는 진전)**
- cap5·cap10 둘 다 0/10은 "마진≠원인"만 증명한다. obs(C8)·**014 §1.2의 margin0 한계주행 동작점(質)**·
  K/scale 미탐색이라는 다른 공통원인이 살아 있어 **데이터가 covariate를 지목하지 못한다(under-determined)**.
- 015 D1이 "obs 동시원인 가능, 반증설계 없이 단정 = 012 A5형"이라 **스스로 적었다**(015:133-136). →
  본문 헤더는 과잉해석이나, D1을 포함해 읽으면 "미입증 작업가설로 강등"으로 읽힌다. **권고: 본문에서도
  결론2를 작업가설로 명시 강등.**

**D2 — "마진이 생존시간 좌우"의 인과가 깨끗한가? → 확증(015 self-critique 정당)**
- 거리환산 2.61배(§3.3)로 마진 효과는 실재하나 "마진"과 "저속 compounding 지연"은 분리 측정 안 됨.
  "5.5배"는 ~2배 과장. **인과 표현을 "마진은 생존을 ~2.6배 늘리나 완주엔 불충분"으로 정정 권고.**

**D3 — S4 결론의 한계 + value 순서 → 확증**
- S4는 P6 value(all·γ0.99)라 normalizer·γ 오염 → "value 무효"로 못 읽음(정당). 깨끗한 cap10-stats·
  γ0.999 value 미시험도 정당. value를 BC 완주 **후**에 두는 순서는 011 §8.2("value는 prior가 만든 plan
  중에서만 고른다", 011:212)와 정합 — BC가 완주 plan을 못 만들면 value가 고를 대상이 없다. **단** §3.2대로
  "완주>충돌"은 γ민감이라 결론2 보강 근거로는 약함.

**D4 — 노이즈 주입이 margin0 cap10에서 복구를 만드나? → 확증 + 증폭 (가장 강한 반박)**
- max\|steer\|=0.4189는 **우연이 아니라 env 하드 clip**(f1tenth_env.py:35 `S_MIN,S_MAX=±0.4189`, :322
  `np.clip(action[0],S_MIN,S_MAX)`)이고 cap10이 **7.76% step에서 이 한계를 명령**한다(§3.1). 복구=
  "더 꺾어 라인 복귀"인데 포화 구간엔 한계방향 노이즈=clip 무효(추가 조향권한 0)·반대방향=표류 악화.
  → **margin0 정책으로는 복구 데모를 만들 권한이 구조적으로 없다.** "완주율 유지 ∧ covariate 확대"가
  동시 성립하는 σ 윈도는 **사실상 존재하지 않을 공산**. 015 D4가 위험을 적었으나 "윈도가 존재하긴 하나
  좁다"로 과소평가 — critic은 "윈도 부재"에 가깝다고 본다. **노이즈주입 폐기, DAgger/중간캡 대체 권고.**

**D5 — 시작점 랜덤화 실현성 → 부분반박 (015가 "바로 가능"을 과대평가)**
- 물리 env는 임의 pose 지원(`f110_env.py:692-727 reset(poses)`; dreamer `f1tenth_env.py:282-291
  options['pose']`). **그러나 실제 수집경로는 통로가 끊겨 있다**: `collect_crash_data.py:128/224`가
  `env.reset()`(무인자) 호출 → 어댑터 `vendor/dreamerv3-torch/envs/f1tenth.py:126-128 F1Tenth.reset(self)`
  가 인자 없이 `self._env.reset()` → pose 전달 불가. **015가 근거로 든 `measure_gap_follower.py:63
  env.reset(poses)`는 base F110Env 직접 경로로, 수집경로와 다른 스크립트다.** → "환경 제약 아님"은 맞으나
  "저비용 글루로 바로 실현"은 거짓: 어댑터 reset 시그니처 수정(코어 인접 글루)이 선행돼야 한다. 또
  free-space/heading 검증은 `extract_centerline.py:163-202`에 있으나 **START_POSES 고정값 전용·강결합**
  이라 임의 pose용 `is_free(x,y)` 재사용 API가 아니다 — 추출 작업 필요. 임의 pose는 벽끼임 후 collision
  종료 + arclength progress 오염(`f1tenth_env.py:300-304`) 위험.

**D6 — 목표 정합(stretch vs 완주) → 확증**
- covariate로 BC 완주해도 56s "모방"이라 <56s stitching 입증 아님(013 C9, 011:126-129·225-227 일치).
  **성공기준을 보고서에서 분리 고정 권고**: (가) baseline 107s 초과 안전완주 = **모방**(Phase2 산출) /
  (나) <56s = **stitching**(Phase3, 011이 "확률 낮음"으로 인정). (가)를 (나)로 포장하면 과잉.

**D7 — 저비용 obs 프로브를 Phase1보다 먼저? → 확증 (가장 중요한 계획 결함)**
- covariate·obs가 식별불가인데 Phase1(다양화수집)은 **covariate가 참일 때만 보상받는 비대칭 베팅**이고
  어댑터 수정까지 필요(D5). 반면 obs 프로브는 **`F1TENTH_LIDAR_DOWNSAMPLE=256` env var flip +
  동일 cap10 30ep BC 재학습 1판(~1-2h, 코어·어댑터 무변경)**으로 한 갈래를 깔끔히 가른다(완주↑→obs
  공간해상도 병목 / 불변→obs 배제·자원을 covariate/質로). **GPU 수~수십시간 수집 전에 이걸 먼저 돌리는
  게 정보경제상 명백히 우월.** 015도 D7에 적어놓고 Phase4로 미룬 모순.

**D8 — V1 페널티 노브의 비약 → 확증**
- "−50에서 충돌윈도 88% 음전환"은 **offline RTG 통계**다(§3.2). value가 학습으로 근사하고 closed-loop
  에서 실제 회피로 잇는지는 별개(011 §9.3 "closed-loop에서 실제로 미는지는 실험으로만 확정" 자인). 또
  이 페널티 발견은 011 §9.1 #1("충돌임박은 모든 γ서 저평가")의 **재확인**이라 신규성도 낮다. S1(Phase3
  옵션)을 띄울 때 이 비약을 시험으로 닫아야.

**D9 — offline RL 정합 → 부분반박 (no-expert 무사, offline 정신은 긴장)**
- no-expert는 코드상 안 깨짐: 수집 정책은 학습 RL 캡 정책(`collect_crash_data.py:359 load_agent`,
  ckpt `cap10_oschersleben/step_45k.pt`)이지 GapFollower expert가 아님(확증). **그러나 011 §8.2가
  정공법으로 축복한 것은 "중간 캡 정책의 자연 완주 rollout 추가"(011:216-217)이지, "env.reset(poses)로
  분포를 인위 설계 + action 외생노이즈로 표류시켜 복구 라벨링"이 아니다.** 후자는 **DART/interactive
  IL**에 가까워 "고정 데이터셋 offline"의 정의와 상충하고, 011 자신이 line 25("추가수집 없이=과제")와
  §8.2("새 수집이 정공법") 사이에 미해소 모순을 안고 있다. 015 D9도 "수집 분포 인위설계가 취지에 맞는지
  재확인"을 자인(015:171). **보고서에서 "covariate-aware 데이터 수집"으로 정직히 프레이밍 권고**(숨기면
  감점, 명시하면 분석 깊이).

**D10 — 비용·순서, value 병행 vs GPU 단일성 → 부분반박**
- value학습 ∥ Phase1수집은 **모순 아님**(수집은 CPU 기본 `collect device=cpu`, value는 GPU). 그러나
  value학습 ∥ Phase2 prior재학습은 **명백 모순**(둘 다 GPU 7/8GB, 011:155 "순차 강제"). 또 BC 완주 전
  value는 "고를 대상이 없어"(011:212) **논리적으로 선행 무의미**. → "병행"을 "Phase1수집과만 병행, 학습
  단계는 순차"로 정정.

## 6. 새로 발견한 리스크 (015가 안 본 것)

- **NR1 — margin0 "복구 데이터 소스" 역설(D4 확장).** 015 처방의 핵심은 "cap10 정책으로 복구 covariate를
  찍자"인데, cap10은 조향 포화 정책이라 **복구를 시연할 권한이 없다**. 즉 *수집 도구 자체가 고장*이다.
  대안: 복구는 **마진 있는 정책**(중간 캡)이라야 시연 가능 — 그러나 저속이라 최종 목표(baseline 초과)와
  긴장. 또는 **DAgger**(cap10이 표류상태를 *라벨*; 노이즈 불요).

- **NR2 — 제3의 독립 원인: 질(質) 동작점(014 §1.2).** cap10 실패(median 66 step, 1.32초, 한 코너도 못
  통과한 **즉발붕괴**)는 "표류→복구실패(covariate)" 서사와 부정합이다(표류할 시간도 없음). 014 §1.2의
  "margin0 풀스로틀 한계주행"이 더 직접적 방아쇠다. **이 질(質) 원인은 obs 해상도·covariate 폭과 모두
  독립**일 수 있다 — 복구 데이터를 아무리 넣어도 *복제하는 행동 자체가 한계*면 닫힌 루프에서 무너진다.

- **NR3 — 두 실패모드를 한 원인으로 환원(under-determination 증거).** cap10 BC 실패길이 [47~96]과 cap5 BC
  [203~1064]는 **완전 비중첩**이다(§3.3). cap10=즉발붕괴(質/obs), cap5=점진실패+롱테일(covariate 정합).
  015는 서로 다른 두 메커니즘을 "복구 covariate 부재" 하나로 묶었다 → cap5의 모드를 고쳐도 cap10의 모드는
  남아 부분 완주에 그칠 위험.

- **NR4 — obs 가설의 전제가 1차소스상 부분 거짓(015에 양날).** 015/프롬프트는 "133D 단일프레임이라
  동역학(속도·heading-rate) 부재"를 함의하나, **obs의 state5 = [vel_x, vel_y, yaw-rate(ang_z),
  prev_steer, prev_speed]**(f1tenth_env.py:50, 242-250; 실데이터 frame2000=[0.46,−0.13,−0.16,0.55,0.5]
  확인)로 **순간 동역학은 단일프레임에 들어있다**. → "동역학 부재" obs가설은 죽고, 살아남는 obs 경쟁가설은
  **"lidar128 공간해상도 부족" + "절대 heading(yaw) 부재"** 한 갈래로 좁혀진다(여전히 결론2를 반증 못 함).

- **NR5 — 기저 산출물 전체가 걸린 베팅으로 격상됨(015 과소반영).** 011은 BC 안전완주 복원을 "중간 확률"로
  베팅했고(011:124-126) 그 위에 분석을 얹어 강한 보고서를 그렸다. **012가 그 대전제를 0/10으로 반증**했다.
  따라서 다양화 fix는 더 이상 "stretch 보너스"가 아니라 **보고서의 기저 산출물(baseline 초과 완주) 자체가
  걸린 高위험 베팅**이다. 실패 시 정성평가의 "명확한 개선"이 사라지고 "실패 분석"만 남는다 → **D7(저비용
  갈래가르기 선행)의 가치가 결정적으로 커진다.**

- **(역으로 015에 유리한 확인) train-eval 하네스는 정합.** 평가(`plan_f1tenth.py`)와 수집
  (`collect_crash_data.py`)이 **동일 `make_env` + `env.reset()`(무인자 기본 pose)**를 쓰고(데이터 시작
  pose=[0.0702,0.3003,2.7979]=dreamer 기본, §3.1 재계산 일치), obs도 로더의 `_downsample_lidar`를 재사용
  (`plan_f1tenth.py:54-61`), action은 raw m/s로 왕복(v_max=20, cmd_v 9.6 보존). → **0/10 BC 실패는
  하네스 버그가 아니라 진짜 닫힌루프 현상.** 이는 015 진단의 정당성을 (어느 원인이든) 뒷받침한다.

## 7. 권고하는 다음 행동 (제약 위반 없이, 비용순)

1. **[최저비용·코어무변경] obs 갈래 먼저 가르기 (D7·NR4).** `F1TENTH_LIDAR_DOWNSAMPLE=256` env var flip +
   동일 cap10 30ep prior 재학습 1판 → BC(K1/K3) 평가(~1-2h GPU, run_in_background). **완주↑ = obs 공간해상도
   병목 확정**(수집 불요, 가장 값싼 해결) / **불변 = obs 한 갈래 배제 → covariate/質로 자원 집중.** 어댑터
   수정·수집 없이 env var만. (256-BC도 재학습은 필요하므로 "무료"는 아니나 Phase1보다 압도적 저비용·정보양분.)

2. **[저비용·진단] cap10 BC 폐루프 trajectory가 데이터와 "언제부터 갈라지나" 측정 (NR2·NR3).** 평가 1판
   재실행 시 per-step obs/action 덤프 → 데이터 동일 구간과 정렬. **step 1~5 즉발 = 質/obs / 30+ step 추종 후
   이탈 = covariate.** 즉발·점진을 가려 NR2(質) vs covariate를 분리. (로그가 aggregate만이면 덤프 옵션 추가.)

3. **[중비용·1·2 결과 분기] covariate가 살아남으면 — 노이즈주입 대신:** (a) **DAgger**(cap10이 표류상태를
   라벨; NR1 역설 회피, no-expert 무사) 또는 (b) **마진 있는 중간 캡 정책의 시작점만-랜덤화한 자연 완주
   수집**(011 §8.2 정합, action 노이즈 없음). **D4의 margin0 노이즈주입은 폐기.** 단 (a)/(b) 모두
   수집경로 어댑터 `reset(options)` 통로 추가가 선행(D5) — 코어 인접 글루, 사용자 승인 후.

4. **[보고서·무비용] 성공기준 분리 고정 (D6) + 진단 프레이밍 정직화 (D9).** (가) baseline 초과 안전완주=
   모방 / (나) <56s=stitching을 본문에 분리 명시. 다양화 수집은 "covariate-aware 데이터 수집(DART/DAgger
   계열)"로 정직히 표기. 결론2는 "입증된 병목"이 아니라 "1·2로 가를 작업가설"로 강등 서술.

5. **[고비용·토대 확보 후] value(γ0.999) — BC 완주 후 (011 §8.2 순서 유지).** Phase1수집과만 병행(CPU/GPU
   분리), prior 재학습과는 순차(D10). "완주>충돌"은 γ민감(§3.2)이라 보강 근거로 약하게만 인용.

> 우선순위 한 줄: **GPU 수~수십 시간을 covariate에 베팅하기 전에, env var 한 줄로 obs 갈래를 먼저
> 가르라.** under-determined 두 가설을 식별하는 데 드는 비용이 가장 싼 실험이 가장 먼저 와야 한다.
